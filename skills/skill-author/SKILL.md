# Skill Author

Write one Conversion Skills skill, or tighten one that exists, in the house style. Then run the gates and report what they say. This is the meta-skill that keeps a growing library consistent.

## When to use
- You want a new skill drafted from a one-line job and a slug.
- An existing skill is loose: vague description, missing sections, redefined terms, off-voice copy, and you want it sharpened.
- You are about to add a skill and need the two-key frontmatter and the body sections right the first time.
- A reviewer wants a skill checked against the rules before it merges.

## Inputs
- The job in one line: what the skill does and the phrases that trigger it.
- The slug: lowercase, hyphens, the folder name under `skills/`. If unstated, derive it from the job.
- Whether this is heavy or high-stakes (an audit, a money or client decision, anything irreversible). If yes, the discipline sections are required.
- The rules, read every run: `CONTRIBUTING.md` (house style, the gates, the discipline-section list), `BRAND.md` (voice, the blocklist, no em-dashes), `GLOSSARY.md` (the canonical terms to reuse, not redefine).
- For a sharpen: the current `skills/{slug}/SKILL.md` to revise in place.
- A near-neighbor skill or two under `skills/` to match exact section phrasing and the ledger-row format.

## Process
1. Read `CONTRIBUTING.md`, `BRAND.md`, and `GLOSSARY.md`. These are the rules. If a draft disagrees with them, the draft is wrong.
2. Read 2 to 3 existing `skills/*/SKILL.md` near the job to match phrasing, the exact ledger-row columns, and the profile-aware Solo/Team/Agency split where it applies.
3. Settle the slug (lowercase, hyphens). For a new skill the folder is `skills/{slug}/`; for a sharpen, open the existing `skills/{slug}/SKILL.md`.
4. Write the frontmatter, exactly two keys. `name` is the slug. `description` is one line, no colon, no em-dash, what the skill does plus the trigger phrases. A colon or a newline in the description fails the validator in `scripts/sync-skills.sh`.
5. Write the H2 body in order: When to use, Inputs, Process, Outputs, Guardrails, References. Process steps are numbered, concrete, and name exact vault paths. Outputs name the files written and the ledger rows appended.
6. Reuse the canonical terms from `GLOSSARY.md` (Vault, Profile, Ledger, Receipt, Firewall, Draft-only, Confidence, Router, Operator, Gate, Judge, Fan-out). Reference them, do not restate their definitions in the skill.
7. Carry the standing guardrails where they apply: Draft-only on anything outbound or published, the agency Firewall (active client only, never a sibling), provenance (cite sources, never invent a metric), credentials live in `_system/connectors.md` and never in the vault.
8. If the skill is heavy or high-stakes, add the three discipline sections before References: Red flags (4 to 6 self-detectors that the run is going wrong), Verification (a checklist of observable exit criteria, never "seems right"), Rationalizations (a two-column table, the excuse to the reality). `scripts/check-sections.sh` requires these on the tagged list in `CONTRIBUTING.md`; if the slug belongs there, they are mandatory.
9. Sweep the draft for brand violations before saving: zero em-dashes (use commas, periods, parentheses; en-dashes in numeric and date ranges only), zero blocklist words, no foreign product names anywhere. Voice is terse, numbers not adjectives, verbs not nouns.
10. Save to `skills/{slug}/SKILL.md`. Output only the SKILL.md text in the draft, nothing wrapped around it.
11. Run `./scripts/sync-skills.sh`. It validates every skill, regenerates `SKILLS.md` and `docs/commands.md`, and runs the six gates (clean-room, budgets, secrets, firewall, brand, sections).
12. Report the gate results plainly. A green run names the slug and the regenerated files. A red gate blocks the skill: name the failing gate and the exact line, and stop. Do not edit around a red gate to force it green.
13. Leave it as a draft for a human to review and commit. Do not commit.

## Outputs
- `skills/{slug}/SKILL.md`: the new or sharpened skill in the house style, two-key frontmatter and the ordered body sections, plus the discipline sections when heavy.
- Regenerated by the script, not by hand: `SKILLS.md` and `docs/commands.md`.
- The gate report: each of the six gates green or red, and on red the failing gate plus the offending line.
- No ledger row. This skill builds the library, it does not move a business metric, so it appends nothing to `Memory/kpi-ledger.md`.

## Guardrails
- Draft only. The skill writes a file and runs the gates. A human reviews and commits. This skill never commits or pushes.
- A red gate blocks. If any of the six gates fails, the skill does not ship. Report the failure, do not paper over it.
- Two keys only in frontmatter, `name` and `description`. The description is one line, no colon, no em-dash.
- Reuse terms, do not redefine them. The canonical definitions live in `GLOSSARY.md`; a skill that restates them drifts from the rest of the library.
- Brand is enforced. Zero em-dashes, zero blocklist words, no foreign product names on any shipped file. `scripts/check-brand.sh` and `scripts/lint-cleanroom.sh` run inside the sync script.
- Budgets hold. Root `CLAUDE.md` 150 lines or fewer, folder routers 60 or fewer, context docs 150 or fewer. A SKILL.md that bloats moves detail into `skills/{slug}/references/*.md`.
- Credentials never enter a skill file. Connectors are named, and configured only in `_system/connectors.md`.

## Red flags
- The description carries a colon or runs to a second line, which the validator rejects before any gate runs.
- A term from `GLOSSARY.md` (Ledger, Firewall, Receipt, Draft-only) gets redefined inside the skill instead of referenced.
- Process steps name no vault path, so the skill reads as advice rather than a runnable procedure.
- The skill is an audit or a client or money decision, yet the discipline sections are missing.
- An em-dash, a blocklist word, or a foreign product name survives into the saved file.
- You edit a passing-looking skill to route around a red gate rather than fixing the cause the gate named.

## Verification
- [ ] Frontmatter has exactly two keys, `name` and `description`; the description is one line with no colon and no em-dash and includes trigger phrases.
- [ ] Body H2s appear in order: When to use, Inputs, Process, Outputs, Guardrails, References.
- [ ] Every Process step is concrete and names exact vault paths; Outputs name files and any ledger rows.
- [ ] Canonical terms are reused from `GLOSSARY.md`, not redefined.
- [ ] Heavy or high-stakes: Red flags, Verification, and Rationalizations are present before References.
- [ ] A brand sweep found zero em-dashes, zero blocklist words, and no foreign product names.
- [ ] `./scripts/sync-skills.sh` ran; all six gates are green; `SKILLS.md` and `docs/commands.md` regenerated.
- [ ] The skill was left as a draft; nothing was committed or pushed.

## Rationalizations
| Rationalization | Reality |
|---|---|
| "The description reads better with a colon." | The validator splits on the colon and fails the skill. One line, no colon, or it never reaches a gate. |
| "I'll define the term inline so the skill stands alone." | `GLOSSARY.md` owns the term. A second definition drifts and the library stops agreeing with itself. Reference it. |
| "The gate is red on a brand nit, I'll just remove the word the linter flags and move on." | Fix the sentence, not the linter. Routing around a gate is how the blocklist gets back into the shipped surface. |
| "It is a small skill, it does not need the discipline sections." | If it audits, decides money, or touches a client, the tagged list requires them. Small is not the test, stakes are. |
| "I'll commit it since the gates passed." | Draft only. A human reviews and commits. This skill stops at green gates and a saved file. |
