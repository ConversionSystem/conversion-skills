# n8n Workflow

Turn a described automation into an importable n8n workflow JSON, with the trigger, nodes, and data mapping spelled out. Draft only, credentials stay out of the file.

## When to use
- You describe a process ("when a form comes in, enrich it, then post to Slack") and want it built in n8n.
- You want a starting workflow you can import, then wire your own credentials into.
- You are moving a manual SOP into an automated run and need the node graph drafted first.

## Inputs
- The workflow in plain words: what starts it, what happens in order, where it ends.
- The trigger type (webhook, schedule, form, app event, manual).
- Source and destination apps or services, plus the fields that move between them.
- Optional: an existing SOP under `Operations/sops/` or a brief under `Projects/{slug}/brief.md` to read the steps from.

## Process
1. Pick the slug. If a project exists, reuse it. Otherwise create `Projects/{slug}/` from the workflow name (kebab-case, e.g. `lead-intake-to-slack`).
2. Read context if present: `Projects/{slug}/brief.md`, any matching `Operations/sops/*.md`, and `Company/stack.md` to confirm which apps are in play.
3. Map the flow. List each step as a node: the trigger first, then transforms (Set, Code, IF, Switch, Merge), then the action nodes (HTTP Request, app nodes, Slack, email). Number them in run order.
4. Define the trigger node. State the type and its config: webhook path and method, cron expression, or form fields. For webhooks, note the test URL is generated by n8n at import, not stored here.
5. Write the data mapping. For every node, list the input fields it reads and the output fields it writes, using n8n expression syntax (e.g. `{{ $json.email }}`, `{{ $node["Webhook"].json.body.name }}`). Keep one mapping table per node.
6. Handle credentials by reference only. Where a node needs auth, name the credential slot (e.g. "Slack account", "HTTP Header Auth") and leave the value empty. Record which connectors are required in `_system/connectors.md`, never the secrets.
7. Add error handling. Note the retry setting per action node and whether a failed run routes to an error branch or a notification. Default outbound and publishing nodes to disabled so the import does not fire on first run.
8. Build the importable JSON. Produce a valid n8n workflow object: `name`, `nodes` (each with `id`, `name`, `type`, `typeVersion`, `position`, `parameters`), `connections`, and `settings`. Use placeholder credential names, no IDs, no tokens.
9. Write the files (see Outputs). Mark the workflow draft and outbound nodes disabled.
10. Write the run notes: how to import (`n8n` then Import from File), which credentials to attach, which nodes to enable, and how to test with a sample payload.

## Outputs
- `Projects/{slug}/final/n8n-workflow.json` · the importable workflow, credentials referenced by name only.
- `Projects/{slug}/workflow-map.md` · the node list in run order with one data-mapping table per node.
- `Projects/{slug}/run-notes.md` · import steps, credentials to attach, nodes to enable, sample test payload.
- `_system/connectors.md` · appended row naming each required connector and its auth type (no secrets).
- `Memory/kpi-ledger.md` · one APPEND-ONLY row when the workflow goes live, e.g. `| {date} | runs/week (n8n {slug}) | 0 | 0 | {target} | n8n executions | low | drafted, not yet enabled |`.

## Guardrails
- DRAFT-ONLY. The JSON is a draft. Outbound and publishing nodes ship disabled so an import cannot send or post on its own.
- Credentials never in the file. Reference credential slots by name only. Secrets live in n8n, tracked as connectors in `_system/connectors.md`, never in the vault.
- PROVENANCE. Map only fields that exist in the described source. Do not invent payload fields or endpoints; mark anything assumed as `TODO: confirm`.
- VOICE from `Library/styles/brand-voice.md` for any node that drafts user-facing text (email, Slack message).
- FIREWALL (agency). When the workflow is for a client, write under `Clients/{slug}/` and touch only the active client.

## References
- `Operations/sops/` for the manual process being automated.
- `Company/stack.md` for the apps in use.
- `_system/connectors.md` for connector and auth tracking.
